Here’s a concrete refactor plan that keeps your *synthetic* intent but makes the Lean layer simpler, more modular, and avoids the “discontinuous argmax” trap.

I’ll phrase it as: **(A)** what to change, **(B)** why, **(C)** what the Lean skeleton looks like.

---

## 0) Guiding principle

Separate three things that are currently entangled:

1. **Geometry**: “simplex-ish” object + `mix` + betweenness/order on edges
2. **Preferences**: ordinal utility + monotonicity along unilateral mixing
3. **Existence/selection**: a *choice* axiom (IVT/Brouwer-like) that picks fixed points

Then prove Nash by *selection*, not by argmax.

---

## 1) Replace `SyntheticInterval` with a standard order + one `mix` axiom

### A. Change

Instead of `PartialOrder + le_total`, just extend `LinearOrder` (non-decidable is fine) and keep only what’s genuinely synthetic:

```lean
class SyntheticInterval (I : Type) extends LinearOrder I where
  mix : I → I → I
  mix_left  : ∀ {x y}, x < y → x < mix x y
  mix_right : ∀ {x y}, x < y → mix x y < y
  mix_idem  : ∀ x, mix x x = x
  mix_comm  : ∀ x y, mix x y = mix y x
```

### B. Why

You instantly inherit a ton of lemmas (`lt_of_le_of_ne`, `le_total`, etc.) and reduce custom proof friction.

---

## 2) Make the simplex carrier an *inductive term algebra* (no axiomatic eliminator)

### A. Change

Replace the axiomatic `carrier + induction` with an inductive syntax of “formal mixtures”.

```lean
inductive SimplexTerm (V : Type) : Type
| vertex : V → SimplexTerm V
| mix    : SimplexTerm V → SimplexTerm V → SimplexTerm V
deriving DecidableEq  -- optional
```

Then define your simplex as just `SimplexTerm` with notation.

If you still want equations like commutativity/idempotence, don’t bake them into definitional equality yet. Either:

* keep them as *rewrite lemmas* in a separate “theory of mix”, or
* later quotient by the congruence (but delay that as long as possible).

### B. Why

Your current `induction` axiom is essentially “free algebra” + extra equations. Lean hates “free + equations” as axioms because you’re implicitly postulating an initial object in an equational theory without constructing it. With an inductive type, the eliminator is real and you keep constructivity.

---

## 3) Define “edge” as *the interval generated by two vertices* (not a big structure)

### A. Change

Instead of your heavy `Edge` structure, define:

```lean
inductive EdgeTerm (a b : V) : Type
| left  : EdgeTerm a b
| right : EdgeTerm a b
| mix   : EdgeTerm a b → EdgeTerm a b → EdgeTerm a b
```

and an interpretation map into the simplex:

```lean
def EdgeTerm.toSimplex : EdgeTerm a b → SimplexTerm V
| .left  => .vertex a
| .right => .vertex b
| .mix x y => .mix (toSimplex x) (toSimplex y)
```

Then you can *postulate* an instance `[SyntheticInterval (EdgeTerm a b)]` if you want interval structure on edges, or define it syntactically (e.g. by taking `EdgeTerm a b` as your “interval object” and axiomatize its order + `mix` there).

### B. Why

This matches your pedagogy: an edge is literally “formal mixtures of endpoints”. Much easier to reason about and to generate the “finitely presented fragments”.

---

## 4) Refactor utility: make it only talk about *comparisons* and the monotone-mix axiom

### A. Change

Instead of `ExtendedUtility` storing a function `carrier → U`, store only a relation capturing “weakly preferred” (or use `Preorder` on outcomes).

For each player `i`, define a preorder on outcomes and then utility is just:

```lean
class Utility (G : Game) (i : Player) where
  le : Outcome → Outcome → Prop
  -- preorder axioms for le
```

Then you add the one synthetic property you really use:

> along unilateral mixing on the *strategy representation*, the outcome preorder is monotone.

This becomes:

```lean
axiom unilateral_mix_monotone
  (s r : Profile) (i : Player)
  (h_same : ∀ j, j ≠ i → s j = r j)
  (h_ord : outcome s ≤ᵢ outcome r) :
  outcome s ≤ᵢ outcome (profileMix s r i) ∧
  outcome (profileMix s r i) ≤ᵢ outcome r
```

(Where `profileMix` mixes only player `i`’s component.)

### B. Why

This keeps everything ordinal and prevents accidental dependence on cardinal structure later.

## 5) Kill discontinuous argmax: define best-response via *crossing selection*, not maximization

This is the key move.

### A. Change

Stop trying to define `F : Simplex → Simplex` as “take best response” by searching maxima. That’s what becomes discontinuous.

Instead define, for each player `i`, a **choice operator** that, given:

* an edge `e : EdgeTerm a b` for player `i` (two candidate actions),
* a fixed opponent profile `p_-i`,
* a predicate expressing “prefers left vs right” that flips between endpoints,

returns a point on the edge where the player is “indifferent” (or “balanced”) in your synthetic sense.

So introduce a minimal *1D* axiom:

```lean
class CrossingChoice (I : Type) [SyntheticInterval I] where
  choose :
    (f : I → Bool) →
    f left = true → f right = false →
    {x : I // f x = true ∧ f (mix x right) = false}  -- or your preferred crossing witness
```

(Replace `Bool` with your two-element object; or use Prop and require both sides.)

Then define “player i response to opponents” as:

* choose a point on some edge(s) using `choose` applied to preference flips,
* or iterate over a finite cover of the simplex by edges if needed.

### B. Why

This makes the Nash proof trivial: your “fixed-point choice axiom” directly applies to a continuous-by-axiom selection map.

It also matches your narrative: *we hand-pick the first iterated mix landing in the crossing region.*

---

## 6) Prove Nash existence as “Brouwer-choice on the product of edges”

### A. Change

State Nash existence as a theorem that uses only:

* finite action sets,
* monotonicity-under-unilateral-mix,
* `CrossingChoice` (1D),
* and one multi-D axiom that you can either postulate or derive:

```lean
class FixedPointChoice (Δ : Type) where
  choose_fp :
    (F : Δ → Δ) →
    (crossing_condition F) →
    {x : Δ // F x = x}
```

Then prove Nash existence by constructing `F` via the crossing-choice “response” operators, rather than argmax.

### B. Why

You avoid upper hemicontinuity / Kakutani / all the classical headaches, while staying faithful to your synthetic plan.

---

## 7) Minimal concrete “stress test” target in Lean

After this refactor, aim to formalize in this order:

1. **Edge monotonicity lemma** (your current `mixBetweenness` style)
2. **Best response contains pure** (still true, but now it will be an induction on `SimplexTerm`)
3. **1D crossing lemma** (your IVT-style axiom interface)
4. **Toy Nash existence for 2×2** using only 1D crossing + finite search over edges
5. Only then: **general finite Nash** using your multi-D fixed-point axiom

This prevents you from rebuilding half of topology in Lean before you have a working demo.

---

## 8) What this buys you immediately

* Proof terms become *tiny* (mostly `simp` + `cases` + apply axiom).
* Models become easy: you can instantiate `SimplexTerm` with concrete `V` and interpret `mix` however you want (or keep it uninterpreted).
* Your “finitely presented” style becomes literal: subterms are finite trees.
* Later, when you add “program equilibrium”, you can reuse the exact same infrastructure: programs are just another finite action set (codes), and your extra computability axioms sit in a separate layer.
